I'm going to explain the very basics of automations here.

What?

Yeah, you're right, I should start with what is an automation and why would I want one? 

An automation is how you get Home Assistant to do things when something happens. That something might be a time, or device in your home doing something, or anything else that HA can detect.

It's very much WTDT - When This, Do That.

A close relative of automations are scripts. The main difference is that scripts have to be called by something else (such as an automation).

Entities

Before we go on to automations, I should touch on the automation relevant parts of entities.

State

Automations have to be on (enabled) to run. If you turn an automation off you disable it, and it won't run.

This can be useful once you're doing some more complex logic, but for now just remember that if your automation doesn't run, check that it's turned on.

Let's start building the automation, as if it was going in automations.yaml:
- initial_state: on

About the automation

All automations should have an alias - the friendly name for you so you remember what it does. If your automation doesn't have an id (see below) then the alias is used to create the entity_id of the automation.

The UI editor needs a unique id. The editor itself uses the Unix timestamp format, but you can use anything you want, as long as it only contains valid characters. In short, that's lower case (ASCII) letters, numbers, and the underscore.

If you want to force an automation to be enabled or disabled on startup (or when you reload automations) regardless of their previous state, you can set the initial_state.

You can also add a description here, if you want more details than the alias can provide.

Let's add to our automation
- initial_state: on
  alias: "My first automation"
  id: my_first_automation
  description: The very first automation I wrote

Trigger

Now we're getting to the useful bits. Triggers are what starts an automation:

When any of these becomes true

If you have five different triggers, then any of them becoming true (firing) will start the automation. If that trigger remains true it won't cause the automation to run again.

There are many different trigger types, but the ones that are most commonly used are probably state, and numeric state.

State

A state trigger fires when the entities state changes, or the state of a specified attribute. You can require it to change to, or from, a particular value, and you can require it to have changed to that value for a given amount of time.
