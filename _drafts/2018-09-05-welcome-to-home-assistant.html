Where do you begin? What do you need to know?

In the beginning

Home Assistant is an event system - things happen, and those generate events. If a sensor reads a new value, that's an event. The time changes, that's an event. Of course, that's all well and good for what is happening, not what has happened. For that, Home Assistant uses state objects, and those state objects are associated with an entity. Those entities are unique (so there can only be one called sensor.flood), and are used in your automations, scripts, scenes, etc.

So what?

Well, when you're writing that automation, triggers fire based off of events (though you can use states to check the previous value too), and conditions use states. Triggers are for this just happened and conditions for this is currently true.

This is why triggers are always or statements - only one trigger has to be true. Conditions can be either and statements or or statements, because they're checking historical information.

Keep that in mind as you write your automations.

Entities

That entity (sensor.flood) we mentioned consists of two parts, the platform (sensor) and the name (flood). Sometimes you'll see the platform referred to as the domain.

Entities are unique, and entities are normally lower case. If you define a sensor that you think will be sensor.Front_door then instead you'll find it as sensor.front_door. It's that entity ID that you'll use in automations, and templates, and everywhere else.

You can also give entities a friendly name for the UI. That's just for your convenience and you can call every entity in your system Fred should you want to.

Configuration file format

The configuration file uses YAML. YAML can be hard to learn at first, but it's basically really a really simple format:
header: # This is a comment
  sub: # This entity will be header.sub
    key: value
    other: list, of, values
    another:
      - list
      - of
      - values
  sub2: # This entity will be header.sub2
    key: value
or:
header2:
  - name: myThing # This entity will be header2.mything
    key: value
    other: list, of, values
    another:
      - list
      - of
      - values
  - name: other_thing # This entity will be header2.other_thing
    key: value2
    other: list, of, values
    word: another
The first is the format used by alert, group, input_boolean, input_select, input_text, input_number, input_datetime, and intents (and probably a few others I've forgotten). The other is used by most things, like automations, sensors, etc etc.

Any header must only appear once. You can't have two sensor: lines (outside of packages). If you really want more then you have to add something to make it unique - for example the second might be sensor two: or sensor front_door:. The thing you add doesn't matter, it's not used anywhere, it just has to be unique.

The only other important thing to know is how to indent. Don't use the tab key, and be consistent with the number of spaces you use per indent. Two spaces per indent level are the standard advised, but if you want to use three, or four, or one, that's up to you. The problem of course is that it makes it much harder to include other's work if you don't use two.

Expanding the capabilities

Components are where the flexibility, and complexity, of Home Assistant arrives. At the time I wrote this there were 39 sections, containing 1,146 components, plus a whole host of unofficial custom components.

Components belong to a platform. Those platforms are things like switch, or sensor, or device_tracker. You add the component under a header for that platform, and the entities created will start with that platform. There are a few (those in the other category) are their own platform.

For example, here's the random component for the sensor platform:
sensor:
  - platform: random
    name: rainbow
Now we'll have an entity called sensor.rainbow that we can use in automations. If we want another random sensor, we'll need to give it another name (or let the system handle it for us).

Custom components have one problem though. They're not always updated to account for the changes in Home Assistant, so sometimes you'll upgrade Home Assistant and then things go wrong. If that happens to you, disable any custom components and see if that fixes your problems.
